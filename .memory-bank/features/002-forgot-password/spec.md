# Specification: Система сброса пароля

**Статус:** Active  
**Версия:** 1.2  
**Related Issue:** [GitHub Issue #2](https://github.com/OlegPhenomenon/video-chat-and-translator/issues/2)  
**Тех. стек:** Ruby on Rails, Inertia.js, React, PostgreSQL, Devise, Redis (очереди), Sidekiq, RSpec.

## 1. Цель

Дать зарегистрированным пользователям с **подтверждённым email** возможность восстановить доступ к аккаунту при утере пароля. Пользователи **без** подтверждённой почты не должны иметь возможности пройти сценарий восстановления пароля (функция для них недоступна).

### Политика anti-enumeration (зафиксировано)

При запросе сброса пароля **всегда** показывается **одно и то же** пользовательское сообщение об успешном приёме запроса (текст из `i18n`), независимо от того, существует ли пользователь с таким email, подтверждена ли почта или письмо реально ушло в очередь. Это обязательное требование; отдельные сообщения для «email не найден» или «сначала подтвердите почту» **не** показываются на этом шаге.

## 2. Зависимости и контекст

- Реализация опирается на завершённую (или выполняемую параллельно) систему авторизации из **001-auth-email**: Devise, `:confirmable`, Inertia-страницы входа/регистрации, фоновая отправка писем, русская локализация UI.
- В спецификации 001 восстановление пароля явно вынесено в out of scope — настоящий документ его закрывает.

## 3. Scope

### Что делаем

- Модуль Devise **`:recoverable`**: запрос сброса по email, одноразовый токен, срок жизни токена, установка нового пароля по ссылке из письма.
- Политика доступа: письмо со ссылкой сброса уходит **только** если пользователь существует **и** `confirmed_at` заполнен (почта подтверждена). Для несуществующего или неподтверждённого email письмо **не** отправляется; UI при этом следует **политике anti-enumeration** (раздел 1), без обхода подтверждения.
- UI: страница «Забыли пароль?» (ввод email), страница ввода нового пароля по токену; стиль форм согласован с 001 (Tailwind, `rounded-md`, форма по центру).
- **Локализация:** весь пользовательский контент страниц сброса пароля, подписи кнопок, сообщений об ошибках/успехе, flash и текстов, показываемых пользователю в этом потоке (включая тексты писем со ссылкой сброса), хранится в **файлах русской локализации** с использованием **i18n** — **без** захардкоженных русских строк в React/ERB, по тому же принципу, что в [001-auth-email/spec.md](../001-auth-email/spec.md) (п. «Весь контент страниц авторизации и регистрации…» и **Localization** в требованиях к реализации).
- Ссылка «Забыли пароль?» на странице входа.
- Отправка письма со сбросом **в фоне** (Sidekiq), по той же логике, что и письма подтверждения в 001.
- **Интеграционные автотесты (RSpec):** покрывают логику запроса сброса, постановку письма в очередь, смену пароля по токену, негативные сценарии (в т.ч. anti-enumeration, неподтверждённый пользователь, просроченный токен, валидация пароля) — в духе формулировки 001: *«интеграционные тесты, которые покрывают все сценарии… с использованием Rspec»*.
- **Доказательства (скриншоты):** для подтверждения работы ключевых сценариев UI создаются скриншоты в каталоге **`screenshots/002-forgot-password`** с помощью утилиты **playwright-cli** — **аналогично** требованию 001 к каталогу `screenshots/001-auth-email` и пункту **«Доказательства»** в критериях приёмки [001-auth-email/spec.md](../001-auth-email/spec.md).

### Что не делаем (Out of Scope)

- OAuth и сторонние identity-провайдеры.
- Смена пароля «в профиле» для авторизованного пользователя (отдельная итерация, если понадобится).
- Кастомные каналы доставки (SMS, push) — только email.

## 5. Инварианты (согласованно с 001)

1. Пароли только в виде хеша (BCrypt через Devise).
2. Сброс пароля не подменяет требование подтверждения email: **неподтверждённый** аккаунт не получает рабочий сценарий восстановления пароля через письмо со ссылкой.
3. Один email — один аккаунт (как в 001).

## 6. Сценарии ошибок и краевые случаи

| Ситуация | Ожидаемое поведение |
| :--- | :--- |
| Email не зарегистрирован | Письмо со сбросом **не** отправлять. UI: **то же** нейтральное сообщение об успешном приёме запроса, что и когда письмо реально уходит (**anti-enumeration**, раздел 1). |
| Email зарегистрирован, почта не подтверждена | Reset-письмо **не** отправлять. UI: **то же** сообщение, что в строке выше; причину (неподтверждённый email) **не** раскрывать. Внутренний лог для поддержки — опционально, без утечки в UI. |
| Невалидный формат email | Клиентская/серверная валидация: ошибка у поля; письмо не отправляется (anti-enumeration на этот кейс не распространяется: пользователь ещё не отправил корректную форму). |
| Новый пароль не проходит правила (как в 001: например **меньше 8 символов**) или пароль и подтверждение **не совпадают** | Сообщения у соответствующих полей (красная рамка + текст под полем), как на странице регистрации в 001; пароль в БД **не** обновляется. |
| Истёкший токен сброса | Сообщение о недействительной или просроченной ссылке; предложение запросить сброс снова. |
| Повторный запрос сброса | Новый токен делает старый недействительным (поведение Devise по умолчанию — сохранить и задокументировать). |
| Ошибка доставки письма (SMTP) | Обработка через очередь (retry по настройкам Sidekiq); веб-запрос не блокируется на SMTP. Пользователь уже получил нейтральное сообщение об приёме запроса; при необходимости тексты про задержку доставки / папку «Спам» — по аналогии с сообщениями в 001 для писем подтверждения (если такой баннер добавляется на шаг запроса сброса, зафиксировать ключ `i18n` в D2). |

## 7. Критерии приёмки эпика (сводно)

1. Подтверждённый пользователь может запросить сброс, получить письмо, задать новый пароль и войти.
2. Неподтверждённый пользователь **не** может завершить восстановление доступа через email-ссылку сброса.
3. UI визуально согласован с формами авторизации из 001; **все** пользовательские тексты вынесены в **файлы русской локализации** (`i18n`), без захардкоженных строк в интерфейсе — в соответствии с подходом [001-auth-email/spec.md](../001-auth-email/spec.md).
4. Письма уходят через фоновую обработку.
5. **Интеграционные** автотесты (RSpec) покрывают основные и критичные негативные сценарии; **все** соответствующие тесты зелёные (как пункт 7 в критериях приёмки 001).
6. Пути и имена маршрутов сверены с актуальным `video_chat_and_translator/config/routes.rb` на момент реализации.
7. **Доказательства:** для ключевых сценариев созданы скриншоты в **`screenshots/002-forgot-password`** через **playwright-cli** — **аналогично** пункту 6 критериев приёмки в [001-auth-email/spec.md](../001-auth-email/spec.md) для `screenshots/001-auth-email`.
8. Работа не считается завершённой, пока не выполнены все требования настоящей спецификации (как пункт 8 в 001).

## 8. Ограничения на реализацию

1. **Аутентификация:** сброс пароля только через Devise **`:recoverable`**; не реализовывать собственные токены и хранение секретов вне Devise.
2. **Очередь:** отправка писем со сбросом — только через фоновую обработку (**Sidekiq** + Active Job), как для писем в 001; не отправлять reset-mail синхронно в HTTP-запросе.
3. **Схема БД:** миграции добавлять **только после согласования** с владельцем репозитория (правило проекта).
