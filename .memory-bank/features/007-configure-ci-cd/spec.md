# Specification: Настроить CI / CD

**Статус:** Draft  
**Версия:** 1.1  
**Related Issue:** [GitHub Issue #7](https://github.com/OlegPhenomenon/video-chat-and-translator/issues/7)  
**Тех. стек:** GitHub Actions, Docker / Docker Compose, Ruby on Rails 8, Kamal, PostgreSQL (в compose), RSpec (планируется в приложении), RuboCop / Brakeman / bundler-audit (уже в Gemfile приложения).

**Источник брифа:** [brief.md](./brief.md)

## Примечание о скоупе и трекинге

Эпики 1 (CI) и 2 (CD) логически независимы: их можно реализовывать и принимать по отдельности. Для удобства трекинга допускается разнести документацию на два артефакта (например `007a-configure-ci` и `007b-configure-cd`); в текущем файле оба эпика сохранены в одном документе из‑за связности с issue #7.

## 1. Цель

Обеспечить команде **единую точку правды** по качеству кода на событиях PR и слияния в `main` (автоматические проверки, включая тесты приложения в среде, согласованной с контейнерной разработкой), а также **один документированный, повторяемый путь** от проверенного состояния кода до развёртывания в staging и production с однозначным критерием успеха и возможностью аудита выкладки.

## 2. Инварианты

Правила ниже действуют для всей реализации по этой спецификации, независимо от фичи.

1. **Секреты:** значения секретов и токенов **никогда** не выводятся в логи CI/CD и не коммитятся в репозиторий.
2. **Паритет образа с dev:** job проверки приложения собирает и использует образ из **`docker/Dockerfile`** с build context **`video_chat_and_translator/`**, как в `docker/docker-compose.yml`. Отличия от compose (переменные, `RAILS_ENV`, команда запуска) допускаются **только** если перечислены в `/docs/ci-parity.md` с обоснованием; скрытого расхождения Dockerfile/образа между dev и CI быть не должно.
3. **Production:** автоматический деплой в production **недоступен** без **обоих** условий: (a) явного approval в GitHub Environment `production` **или** деплой только по **аннотированному git-тегу** релиза — выбранный вариант фиксируется один раз в `/docs/cd-triggers.md` и в workflow; (b) успешного прохождения согласованных проверок на том же коммите/теге, что выкладывается.
4. **Миграции:** не добавлять миграции и не менять схему БД **только** ради CI; исключения — только с явного разрешения владельца репозитория.
5. **Гемы:** не добавлять новые гемы без явного разрешения владельца репозитория (соглашение проекта).

## 3. Ограничения реализации

- **Инфраструктура и рантайм:** автоматизированный агент (CI-бот, LLM-агент) **не** перезапускает прод-серверы, **не** выполняет деплой от имени команды без человека и **не** считается исполнителем операционных шагов в GitHub UI (branch protection, secrets).
- **Миграции и схема БД:** только с согласования владельца репозитория.
- **Новые гемы:** только с согласования владельца репозитория.

## 4. Решения, требующие согласования (зафиксировать до merge PR реализации)

| Решение | Кто фиксирует | Где фиксируется |
| :--- | :--- | :--- |
| Перечень статических проверок в CI (минимум: `rubocop`, `brakeman`, `bundle audit` — все три, **если** не сняты явным решением с причиной) | Владелец репо или назначенный мейнтейнер | `/docs/ci-checks.md` |
| Триггеры CD (staging: push в `main` / вручную / и т.д.; production: approval vs тег) | Владелец репо + мейнтейнер инфраструктуры | `/docs/cd-triggers.md` |
| Использование SSL-terminating `proxy` в Kamal для данного окружения | Мейнтейнер инфраструктуры | Комментарий в `deploy.yml` + раздел в runbook |
| Отсрочка фичи 1.3 (документация branch protection), если защита ветки пока не внедряется | Владелец репо | Запись в `/docs/ci-checks.md` или decision log в `/docs/` с датой |

После записи в таблице **не** используются формулировки «по согласованию» / «при необходимости» в AC — ссылаются на соответствующий документ.

## 5. Зависимости и контекст

- В репозитории уже есть `.github/workflows/ci.yml`: линтинг (actionlint, shell, markdown) и job **smoke** (`make bootstrap`, `scripts/test-ci.sh`) — проверка mise/CLI, **без** прогона тестов Rails в Docker.
- Разработческий контур приложения: `docker/docker-compose.yml`, `docker/Dockerfile`, код в `video_chat_and_translator/`.
- Поставка задокументирована в стеке как **Kamal**: `video_chat_and_translator/config/deploy.yml` (сейчас с плейсхолдерами хостов/registry).
- В `video_chat_and_translator/Gemfile` в группе `:development, :test` уже есть `rubocop-rails-omakase`, `brakeman`, `bundler-audit`; **отдельного** `rspec-rails` и каталога `spec/` в приложении на момент спецификации может не быть — см. §7.

## 6. Scope

### Что делаем

- Job(ы) в GitHub Actions на `pull_request` и `push` в `main`, выполняющие проверки приложения в Docker согласно §2 (инварианты).
- Прогон команд из `/docs/ci-checks.md`; после появления в репозитории `spec/` и RSpec — обязательный `bundle exec rspec` в том же job (или явно описанном дочернем шаге).
- Артефакты и логи при падении — по §8 (эпик 1, сценарии сбоев) и фиче 1.2.
- Документация в `/docs`: воспроизведение CI, `ci-parity.md`, runbook CD, `cd-triggers.md`.
- По плану эпика 2: workflow деплоя и приведение `deploy.yml` к целевым окружениям.

### Что не делаем (Out of Scope)

- Замена Kamal на другой оркестратор без отдельного решения команды.
- Сама по себе настройка branch protection в UI GitHub — операционный шаг; в коде — только документация (фича 1.3).
- Миграции и новые гемы без разрешения владельца репозитория.

## 7. Эпики и фичи (vertical slices)

### Эпик 1 — CI: единая точка правды по качеству (PR + `main`)

#### Сценарии сбоев и ограничения (эпик 1)

| Событие | Ожидаемое поведение |
| :--- | :--- |
| **Сборка Docker-образа в CI завершилась с ошибкой** | Job помечается как failed; диагностика — по логу шага `docker compose build` / `docker build` в GitHub Actions. Дополнительно: при failure загружается артефакт `docker-build-log.txt` (полный stdout/stderr сборки), если шаг записывает лог в файл, **или** в `/docs` описано, что достаточно лога шага в UI (выбрать один вариант в PR реализации и не менять без обновления docs). |
| **Превышен лимит времени job** | У job проверки приложения задано `timeout-minutes` (рекомендация: 45, не более 60 на стандартном GitHub runner без согласования). По таймауту job failed; в логе видно прерывание по времени. |
| **Падение теста / линтера / brakeman** | Job failed; срабатывают deliverables фичи 1.2 (артефакты). |
| **Недоступен registry при pull базовых образов** | Поведение GitHub Actions по умолчанию (retry/ошибка сети); в `/docs/ci-troubleshooting.md` (или разделе в основном CI-доке) — пункт «повторить run / проверить статус Docker Hub / mirror». |

### Эпик 2 — CD: один документированный путь к staging и production

#### Сценарии сбоев и ограничения (эпик 2)

| Событие | Ожидаемое поведение |
| :--- | :--- |
| **Провал `kamal deploy` (exit code ≠ 0)** | Workflow помечается failed; в логе — вывод Kamal. В runbook (фича 2.1) описан порядок действий: сохранить версию/тег образа, выполнить **откат** предыдущей версии командой из runbook; в AC фичи 2.1 проверяется наличие раздела «Откат» с конкретными командами и критерием успеха отката. |
| **Недоступен registry или целевой сервер** | Деплой завершается ошибкой; без «успешного» зелёного статуса при частично выкаченном состоянии workflow не помечается успешным. В runbook — пункт эскалации и проверки сети/SSH. |
| **Два деплоя одновременно** | В workflow деплоя задана политика `concurrency`: одна группа на окружение (например `deploy-staging`, `deploy-production`), **`cancel-in-progress: false`**, чтобы второй запуск ждал завершения первого, а не прерывал его. Текст политики копируется или резюмируется в `/docs/cd-triggers.md`. |
| **Параллельные push в `main`** | CI: уже используется `concurrency` в существующем workflow (сохранить или согласовать с новым app-job). CD: см. строку выше. |

| **Фича 2.1** | Runbook поставки (Kamal) |
| **Ценность** | Предсказуемое число шагов и критерий успеха; меньше устных передач между мейнтейнерами. |
| **Deliverables** | Документ в `/docs` (runbook): предпосылки (registry, SSH, `.kamal/secrets`), команды развёртывания, **отдельный раздел «Откат»** с командами Kamal и критерием успешного отката (приложение отвечает на smoke URL / прежняя ревизия на хосте). Разделы: staging vs production, критерий «релиз успешен», куда смотреть логи, политика секретов (например 1Password). **Чеклист для нового мейнтейнера** (в конце runbook или отдельным файлом `/docs/cd-maintainer-checklist.md`), минимум: (1) доступ к registry и pull образа; (2) SSH на целевые хосты; (3) заполнение секретов из хранилища команды; (4) выполнение dry-run или `kamal config` без ошибок; (5) деплой в согласованное **непродакшен** окружение; (6) smoke по URL из runbook; (7) выполнение **учебного отката** на том же окружении или согласованный tabletop с владельцем репо, зафиксированный в тикете. |
| **Acceptance criteria** | 1) Runbook содержит чеклист из deliverables; все пункты нумерованы. 2) **Метрика готовности документа:** назначенный мейнтейнер (не автор runbook) проходит чеклист **полностью** и фиксирует результат в тикете/issue; успех — **все пункты чеклиста отмечены выполненными без запросов пояснений к автору runbook** (допускается обращение к владельцу репо по доступам, не по содержанию шагов). 3) Раздел «Откат» содержит команды и ожидаемый результат; проверка — реальный учебный откат **или** подписанный tabletop с мейнтейнером (ссылка в тикете). 4) Описаны порядок шагов успешного деплоя, что считается ошибкой, куда смотреть логи. |
| **Grounding (проект)** | `video_chat_and_translator/config/deploy.yml`, `video_chat_and_translator/Dockerfile`, `docs/...` (runbook + чеклист), при наличии — `bin/kamal`, шаблоны `.kamal/` без секретов. |

## 8. Сводка: бриф → фичи

| Формулировка брифа | Фичи |
| :--- | :--- |
| PR + merge в `main`, единый набор проверок, тесты в среде, согласованной с dev | 1.1, 1.2 |
| Прозрачный итог прогона для команды | 1.2 |
| Документация обязательных checks (human gate в GitHub отдельно) | 1.3 |
| Один документированный процесс до staging/production | 2.1 |
| Повторяемость и аудит выкладки | 2.2, 2.3 |

**Фича 1.3** входит в объём эпика 1 **по умолчанию**. Отложить документирование/внедрение protection допускается только с записью в §4 (решения) — иначе таблица выше остаётся без исключений.

## 9. Риски и зависимости

- **Тесты приложения:** до появления `rspec-rails` и `spec/` в `ci-checks.md` фиксируется явный пропуск RSpec с датой; закрытие брифа по «обязательным тестам» — после включения RSpec в набор (отдельная задача или тот же PR — по плану команды).
- **База данных для тестов в CI:** изменения compose и схемы — только с разрешения владельца репозитория (инвариант §2 п. 4).
- **Проверка деплоя:** живой `kamal deploy` и секреты — ответственность мейнтейнера; агент не заменяет человека на инфраструктуре (§3).
