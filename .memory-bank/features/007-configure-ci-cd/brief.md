# Brief: Настроить CI / CD

**Related Issue:** [GitHub Issue #7](https://github.com/OlegPhenomenon/video-chat-and-translator/issues/7)
**Status:** draft
**Date:** 04.04.2026

### Проблема

Сегодня нет общей для команды точки правды о том, проходят ли автоматизированные тесты после изменений в репозитории: проверка остаётся на локальных машинах разработчиков. В результате дефект может оказаться в основной ветке до того, как о нём узнает вся команда, а время на обнаружение регрессии зависит от того, кто и когда локально запустил тесты.

Поставка в целевые окружения не зафиксирована как один воспроизводимый сценарий с предсказуемым порядком шагов и явным результатом. Измеримый эффект для команды: нельзя заранее назвать, сколько ручных действий и на чьей стороне потребуется от мержа до выкладки, и нет единого аудита «кто и что выкатил».

### Для кого

Команда разработки и мейнтейнеры репозитория (нужны прозрачная сборка и предсказуемая выкладка). Косвенно — пользователи сервиса: меньше риска доставить в прод необнаруженную поломку из-за отсутствия обязательного автоматического прогона проверок на общем контуре.

### Контекст

Задача заведена как [GitHub Issue #7](https://github.com/OlegPhenomenon/video-chat-and-translator/issues/7). Сейчас это важно, потому что при росте числа изменений и участников ручная проверка и разрозненные практики выкладки масштабируются хуже, чем код: стоимость ошибки на основной ветке и при релизе растёт. В проекте уже заложены контейнеризованная разработка и выкладка через Kamal (см. описание стека в репозитории) — без оформленного контура проверки и поставки этот стек не даёт команде стабильной операционной дисциплины.

### Желаемый результат

При открытии и обновлении pull request, а также при слиянии в основную ветку (`main`) автоматически выполняется единый набор проверок качества (включая прогон автоматизированных тестов приложения в среде, эквивалентной той, что использует команда для проверки в репозитории). Итог каждого прогона (успех или провал с артефактами для разбора) доступен всей команде без необходимости полагаться только на локальный запуск у одного человека.

Путь от проверенного состояния кода до развёртывания в staging и production — один документированный, повторяемый процесс: порядок шагов и критерий «релиз состоялся / не состоялся» однозначны для команды.
