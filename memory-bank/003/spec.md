# Spec: issue #3

## Источник
- GitHub issue: https://github.com/atamanuk/mapper/issues/3
- Заголовок: `Добавить мэппинг вложенных полей`
- Статус: `OPEN`

## Контекст
Сейчас `Mapper` умеет маппить только прямые пути верхнего уровня, потому что правила собираются для свойств вида `"/Field"` и ищутся по точному совпадению строки пути. Для операций с вложенными путями, например `"/SourceField/SubsourceField"`, такой логики нет.

## Цель
Добавить поддержку мэппинга операций Json Patch для вложенных полей объектов:
- определить соответствие между `SourceField` и `TargetField` по существующим правилам мэппинга `TSource -> TTarget`;
- найти профиль мэппинга для типов вложенных объектов `TSubsource -> TSubtarget`;
- рекурсивно преобразовать вложенную часть пути и значение операции так, чтобы операция над `TSource` корректно превращалась в операцию над `TTarget` для любой глубины вложенности объектных свойств в пределах поддерживаемого JSON Pointer без индексов массивов.

## Что требуется реализовать
1. Научить компиляцию/разрешение правил обрабатывать пути с вложенностью, а не только точные пути верхнего уровня.
2. При мэппинге пути вида `"/SourceField/SubsourceField"` сначала резолвить корневое поле `SourceField`, затем применять мэппинг вложенного профиля для остатка пути.
3. Поддержать вложенный мэппинг для тех же типов операций, которые уже покрыты сейчас: `add`, `replace`, `remove`, `test`.
   Требования к `from` в этой задаче относятся только к внутреннему поведению маппера при наличии непустого значения `from` в операции, независимо от значения `op`.
   В тестах допускается использовать синтетические операции с `op` из списка выше и заполненным `from`, чтобы проверить именно логику маппинга поля `from`.
   Добавлять полноценную поддержку RFC-операций `move` и `copy` в рамках issue не требуется.
4. Сохранить действующие правила:
   - мэппинг одноимённых полей по умолчанию;
   - переименование через `ForMember(...).MapFrom(...)`;
   - `Ignore()`;
   - конвертацию значений при несовпадении типов;
   - вычисляемое преобразование значения из одного исходного поля.
5. Для nested-case поддерживать те же выражения `MapFrom(...)`, которые уже поддерживаются сейчас: выражение должно зависеть ровно от одного исходного поля или вложенного исходного поля. Сценарии, где выражение зависит от нескольких исходных полей, остаются вне объёма задачи.
   Если целевой путь задан явным `MapFrom(...)` от одного исходного вложенного поля, отдельный вложенный профиль для промежуточной пары типов не требуется.
   Явное правило `ForMember(...).MapFrom(...)` имеет приоритет над рекурсивным резолвингом через вложенные профили и над default same-name mapping.
   Если в одном профиле два или более `ForMember(...).MapFrom(...)` приводят к одному и тому же исходному пути, конфигурация считается некорректной и должна завершаться `InvalidOperationException` при компиляции мэппинга с указанием конфликтующего исходного пути.
6. Добавить новый файл тестов для вложенных полей с проверкой обязательных nested-сценариев из раздела "Критерии приёмки".

## Ожидаемое поведение
- Если операция приходит по пути `"/SourceField/SubsourceField"`, результирующий `path` должен указывать на корректный путь в целевой модели, например `"/TargetField/SubtargetField"`.
- Если для верхнего или вложенного целевого поля задан `Ignore()`, операция должна быть пропущена.
- Если тип вложенного значения отличается, должна применяться существующая конвертация/вычисляемое преобразование.
- Если для вложенной пары типов профиль не настроен, `Mapper` должен выбрасывать понятное исключение `InvalidOperationException` с указанием пары типов, для которой не найден вложенный профиль.
  Это требование действует только в сценариях, где для продолжения мэппинга остатка пути нужен именно вложенный профиль, а не прямое правило `MapFrom(...)`, уже задающее полный целевой путь для исходного вложенного поля.
- Если путь или его остаток не может быть сматчен ни явным `MapFrom(...)`, ни default same-name mapping, ни через nested-profile, `Mapper` должен выбрасывать `InvalidOperationException` с указанием исходного пути и пары типов текущего профиля, на котором резолвинг остановился.
- Если операция содержит `from`, значение `from` должно маппиться по тем же правилам, что и `path`.
- Если `Ignore()` срабатывает для `path` или для `from`, операция должна быть пропущена целиком.
- Поддерживаются только пути к leaf-свойствам вложенных объектов. Leaf-свойством в рамках этой задачи считается конечное свойство пути одного из следующих типов: `bool`, `string`, любой enum и любой числовой тип.
- Если путь заканчивается на свойстве другого типа и операция адресует это свойство целиком, такой сценарий считается операцией над целым вложенным объектом, в рамках этой задачи не поддерживается и должен завершаться `NotSupportedException`.
- Поддерживаются только корректные JSON Pointer к вложенным объектным свойствам без массивов и специальных токенов. Неподдерживаемыми и приводящими к `NotSupportedException` в рамках этой задачи считаются:
  - пустой путь;
  - путь без ведущего символа `/`;
  - путь с пустым сегментом, включая `//` и завершающий `/`;
  - путь с массивами или индексами;
  - путь с escape-последовательностями JSON Pointer (`~0`, `~1`).
  Это требование одинаково относится к `path` и `from`.
  Полноценная поддержка таких путей будет реализована отдельно в следующих задачах.
- Проверка unsupported-path выполняется раньше резолвинга правил мэппинга. Соответственно, если `path` или `from` содержит неподдерживаемый JSON Pointer, должен выбрасываться `NotSupportedException` даже в случаях, когда соответствующая операция могла бы быть пропущена из-за `Ignore()`.

## Технические наблюдения по текущему коду
- [src/Mapper/Mapper.cs](/home/me/projects/mapper/src/Mapper/Mapper.cs) хранит скомпилированные мэппинги в `ConcurrentDictionary<TypePair, ICompiledMap>`.
- `CompiledMap.ResolveRule(...)` ищет только полное совпадение строки пути в `_rules`.
- `PatchMapCompiler.BuildDefaultRules(...)` создаёт правила только для прямых свойств верхнего уровня.
- Выражения уже умеют строить вложенные пути через `ExpressionPath` и `SourceMemberDescriptor`, поэтому часть инфраструктуры для nested-path уже есть.
- Текущие тесты находятся в [src/Mapper.Tests/MapperTests.cs](/home/me/projects/mapper/src/Mapper.Tests/MapperTests.cs) и покрывают только верхний уровень.

## Критерии приёмки
- `Mapper` корректно маппит вложенный `path` вида `"/SourceField/SubsourceField"` в целевой путь вида `"/TargetField/SubtargetField"`.
- `Mapper` корректно маппит пути глубже одного уровня вложенности, например `"/A/B/C"` -> `"/X/Y/Z"`, если для каждой вложенной пары типов настроен необходимый профиль или явное правило.
- `Mapper` корректно маппит вложенный `from` по тем же правилам, что и `path`.
- Добавлены тесты для операций, в которых значение `from` должно учитываться маппером.
  Допускаются синтетические операции с `add`/`replace`/`remove`/`test`, если они нужны только для проверки внутреннего маппинга `from`.
- Для вложенных полей работают сценарии:
  - same-name mapping;
  - rename через `ForMember(...).MapFrom(...)`;
  - `Ignore()` на верхнем целевом поле;
  - `Ignore()` на вложенном целевом поле;
  - конвертация значения при отличии типов;
  - вычисляемое преобразование значения из одного вложенного исходного поля.
- Если вложенный профиль `TSubsource -> TSubtarget` отсутствует, выбрасывается `InvalidOperationException`.
- Если корневой сегмент пути, конечный leaf-сегмент или промежуточный remainder пути не удаётся сматчить никаким поддержанным способом, выбрасывается `InvalidOperationException`, а не silent fallback к исходному пути.
- Если путь не требует вложенного профиля из-за явного `MapFrom(...)` от одного вложенного исходного поля, операция маппится без исключения.
- Если один и тот же исходный nested-path может быть сматчен и через явный `MapFrom(...)`, и через рекурсивный nested-profile, применяется явный `MapFrom(...)`.
- Если два `ForMember(...).MapFrom(...)` конфликтуют по одному и тому же исходному пути, компиляция мэппинга завершается `InvalidOperationException`.
- Пути с массивами, индексами JSON Pointer или escape-последовательностями покрыты отдельными негативными тестами и приводят к `NotSupportedException`.
  Отдельным негативным тестом зафиксирован приоритет этой ошибки над `Ignore()`.
- Операции над целым вложенным объектом покрыты отдельным негативным тестом и приводят к `NotSupportedException`.
- Негативные тесты на `InvalidOperationException` проверяют не только тип исключения, но и наличие в сообщении пары типов и проблемного исходного пути или конфликтующего пути конфигурации.
- Добавлен отдельный тестовый файл с nested-сценариями.
- Все тесты проекта проходят через `dotnet test`.

## Риски и вопросы
- В реализации нужно аккуратно разделить:
  - резолвинг корневого сегмента пути в текущем профиле;
  - поиск вложенного профиля для остатка пути;
  - преобразование значения только для операций, где значение допустимо по протоколу Json Patch.
- Поддержка массивов/индексов в JSON Pointer, например `"/Items/0/Name"`, остаётся отдельной задачей.
