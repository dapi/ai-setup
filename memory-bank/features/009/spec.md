# Spec: issue #9

## Источник
- GitHub issue: https://github.com/atamanuk/mapper/issues/9
- Заголовок: `Оформить .prompts`
- Статус: `OPEN`

## Контекст
В репозитории уже есть заготовки промптов в `memory-bank/prompts/`, но они оформлены как внутренние заметки и покрывают только часть workflow по работе с задачами. В issue требуется превратить этот набор в явный паттерн использования prompt-шаблонов через корневую папку `.prompts`, чтобы типовые шаги по подготовке задачи, проверке артефактов и реализации фич выполнялись единообразно.

Issue задаёт CLI-подобный сценарий использования, где prompt выбирается по имени и получает `issue_number`, например:

```text
start_task 5
```

Это означает, что каждый шаблон должен быть оформлен так, чтобы его можно было вызывать по имени файла и чтобы он явно использовал номер GitHub issue как входной параметр.

## Цель
Оформить в корне репозитория каталог `.prompts` с набором шаблонов для типового feature-workflow, где каждый шаблон:
- соответствует конкретному шагу процесса;
- принимает `issue_number`;
- ссылается на реальные артефакты репозитория;
- производит проверяемый результат в `memory-bank/features/{issue_number}/` или относительно уже существующих файлов задачи.

## Что требуется реализовать
1. Создать в корне репозитория папку `.prompts`.
   Она должна быть основной точкой хранения шаблонов промптов для работы над фичами.
2. Перенести или переоформить существующие заготовки из `memory-bank/prompts/` в формат шаблонов в `.prompts`.
   Шаблоны в `.prompts` должны быть самодостаточными и пригодными для прямого использования без необходимости читать старые заметки в `memory-bank/prompts/`.
3. Зафиксировать соглашение использования шаблонов:
   - имя команды соответствует имени prompt-файла без расширения;
   - вторым аргументом передаётся `issue_number`;
   - каждый prompt должен явно использовать переданный `issue_number` в инструкциях.
4. Оформить в `.prompts` следующий обязательный набор шаблонов:
   - `start_task`;
   - `rw_brief`;
   - `mk_spec`;
   - `rw_spec`;
   - `mk_plan`;
   - `rw_plan`;
   - `gnd_plan`;
   - `implement`.
5. Для каждого обязательного шаблона зафиксировать ожидаемое поведение:
   - `start_task`:
     - читает GitHub issue `#issue_number`;
     - создаёт папку `memory-bank/features/{issue_number}/`;
     - строит короткий `brief.md` в этой папке.
   - `rw_brief`:
     - проверяет `brief.md` на полноту, однозначность, непротиворечивость и верифицируемость.
   - `mk_spec`:
     - строит `spec.md` на основе `brief.md`.
   - `rw_spec`:
     - проверяет `spec.md` на полноту, однозначность, непротиворечивость и верифицируемость.
   - `mk_plan`:
     - строит `plan.md` на основе `spec.md`.
   - `rw_plan`:
     - проверяет `plan.md` на соответствие `spec.md`.
   - `gnd_plan`:
     - проверяет `plan.md` на соответствие кодовой базе проекта.
   - `implement`:
     - реализует `plan.md`.

## Требования к содержимому шаблонов
- Каждый шаблон должен явно указывать, какие входные файлы он читает и какой выходной артефакт создаёт или проверяет.
- Все пути в шаблонах должны соответствовать фактической структуре репозитория, в частности использовать `memory-bank/features/{issue_number}/`, а не альтернативные или устаревшие пути.
- Формулировки должны быть достаточно точными, чтобы агент не додумывал:
  - откуда брать исходные данные;
  - какой файл нужно создать или проверить;
  - по каким критериям проводить ревью.
- Проверочные шаблоны (`rw_brief`, `rw_spec`, `rw_plan`, `gnd_plan`) должны ориентировать агента на поиск конкретных дефектов, а не на абстрактный обзор.
- Генерирующие шаблоны (`start_task`, `mk_spec`, `mk_plan`) должны требовать сохранения результата в конкретный файл, а не только вывода текста в чат.
- Шаблоны не должны ссылаться на несуществующие роли, команды, директории или инструменты.

## Ожидаемый workflow
1. Пользователь вызывает `start_task {issue_number}`.
   Результат: создана папка задачи и подготовлен `brief.md`.
2. Пользователь вызывает `rw_brief {issue_number}`.
   Результат: brief проверен и при необходимости скорректирован.
3. Пользователь вызывает `mk_spec {issue_number}`.
   Результат: создан `spec.md`.
4. Пользователь вызывает `rw_spec {issue_number}`.
   Результат: spec проверен и улучшен.
5. Пользователь вызывает `mk_plan {issue_number}`.
   Результат: создан `plan.md`.
6. Пользователь вызывает `rw_plan {issue_number}` и `gnd_plan {issue_number}`.
   Результат: план проверен как на соответствие spec, так и на согласованность с кодовой базой.
7. Пользователь вызывает `implement {issue_number}`.
   Результат: план реализуется в кодовой базе.

## Ограничения и решения
- Нужно опираться на фактическое состояние репозитория и существующий процесс через `memory-bank/features/`, а не вводить параллельную структуру артефактов.
- `.prompts` должна хранить именно шаблоны промптов, а не подменять собой `memory-bank`: проектовый контекст и спецификации остаются в `memory-bank`.
- Если существующие заготовки из `memory-bank/prompts/` неполные или используют другой нейминг, их нужно привести к единому соглашению, а не переносить дословно.
- Плейсхолдер для номера задачи должен быть единообразным во всех шаблонах; нельзя смешивать варианты вроде `{task_number}` и `{task-number}`.

## Критерии приёмки
- В корне репозитория существует папка `.prompts`.
- В `.prompts` присутствуют файлы для всех обязательных шаблонов:
  - `start_task`;
  - `rw_brief`;
  - `mk_spec`;
  - `rw_spec`;
  - `mk_plan`;
  - `rw_plan`;
  - `gnd_plan`;
  - `implement`.
- Каждый шаблон явно использует `issue_number` как входной параметр.
- `start_task` ориентирует на чтение GitHub issue `#{issue_number}`, создание `memory-bank/features/{issue_number}/` и построение `brief.md`.
- `mk_spec` ориентирует на создание `spec.md` из `brief.md`.
- `mk_plan` ориентирует на создание `plan.md` из `spec.md`.
- `rw_brief` и `rw_spec` требуют проверки на полноту, однозначность, непротиворечивость и верифицируемость.
- `rw_plan` требует проверки соответствия `plan.md` спецификации.
- `gnd_plan` требует проверки соответствия `plan.md` фактической кодовой базе проекта.
- `implement` ориентирует на реализацию `plan.md`.
- Содержимое шаблонов не противоречит текущей структуре репозитория и workflow через `memory-bank/features/`.

## Риски и открытые вопросы
- В issue задан пример вызова `start_task 5`, но не зафиксирован конкретный раннер этих шаблонов; поэтому в рамках задачи важно оформить сами prompt-файлы и соглашение их использования, не привязываясь к несуществующему исполняющему инструменту.
- Если в проекте останется и `memory-bank/prompts/`, и `.prompts`, нужно явно решить, что из этого является каноническим источником шаблонов, иначе процесс быстро разойдётся.
