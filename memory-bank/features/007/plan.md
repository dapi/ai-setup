   - Использовать как источник правды текущие файлы:
     - `PROJECT.md`;
     - `src/Mapper/Mapper.csproj`;
     - `src/Mapper.Tests/Mapper.Tests.csproj`;
     - `src/Mapper/Mapper.cs`;
     - `src/Mapper/MapProfile.cs`.
   - Подтвердить, что:
     - исходный код и solution лежат в `src/`, а не в `/scr`;
     - библиотека таргетит `netstandard2.1`;
     - публичные сущности проекта — `Mapper` и `MapProfile`;
     - тесты находятся в отдельном проекте на `xUnit`.

2. Создать раздел `memory-bank/project/`.
   - Добавить `overview.md` с описанием:
     - назначения библиотеки;
     - целевой платформы;
     - основной доменной задачи: преобразование `JsonPatchDocument<TSource>` в `JsonPatchDocument<TTarget>`;
     - ключевых публичных сущностей `Mapper` и `MapProfile`.
   - Добавить `build.md` с описанием:
     - канонической структуры репозитория;
     - того, что сборка выполняется стандартными CLI-командами `dotnet`;
     - расположения solution в `src/Mapper.sln`;
     - отсутствия отдельного build-скрипта, если он действительно не нужен для обычной сборки.
   - Добавить `testing.md` с описанием:
     - тестового проекта `src/Mapper.Tests`;
     - поддерживаемых сценариев тестирования на основе текущих тестов;
     - команды запуска `dotnet test`.

3. Создать `memory-bank/index.md` как стартовый файл.
   - Добавить аннотированные ссылки на:
     - `project/overview.md`;
     - `project/build.md`;
     - `project/testing.md`.
   - Для каждой ссылки использовать формат аннотации из spec:
     - `Что:` краткое описание содержимого файла;
     - `Читать, чтобы:` когда и зачем к нему обращаться.
   - Сделать индекс пригодным как первую точку чтения для нового человека или агента.
   - Не индексировать `features/*`, так как spec этого не требует.

4. Перенести материалы задачи `003` в новую иерархию.
   - Создать `memory-bank/features/003/`.
   - Переместить без изменения содержимого:

5. Обновить `AGENTS.md`.

     - `читай memory-bank/index.md`
   - Проверить, что после сокращения `AGENTS.md` критичный контекст уже доступен через `memory-bank/index.md` и документы в `memory-bank/project/`.

6. Провести итоговую валидацию.
   - Проверить структуру `memory-bank` относительно spec.
   - Проверить, что новые документы не повторяют ошибочное упоминание `/scr`.

   - Проверить, что ссылки в `memory-bank/index.md` рабочие.

## Порядок выполнения
1. Сначала подготовить `memory-bank/project/overview.md`, `build.md`, `testing.md`.
2. Затем создать `memory-bank/index.md`.
3. После этого перенести `memory-bank/003` в `memory-bank/features/003`.
4. В конце сократить `AGENTS.md`.

Такой порядок нужен, чтобы к моменту упрощения `AGENTS.md` новый вход через `memory-bank/index.md` уже существовал и содержал основной контекст.

## Риски
- В `PROJECT.md` упоминается кодогенерация, но текущая реализация библиотеки выглядит как runtime-компиляция маппингов. При подготовке `overview.md` нужно не зафиксировать спорное утверждение как подтверждённый факт, если оно не видно в коде.
- В документации нельзя обещать несуществующие build-скрипты или альтернативную структуру репозитория. Опорой должно быть текущее состояние `src/Mapper.sln`, `Mapper.csproj` и `Mapper.Tests.csproj`.
