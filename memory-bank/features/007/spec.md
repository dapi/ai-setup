# Spec: issue #7

## Источник
- GitHub issue: https://github.com/atamanuk/mapper/issues/7
- Заголовок: `Построить memory-bank`
- Статус: `OPEN`

## Контекст
Сейчас в репозитории `memory-bank` почти пустой: помимо документов текущей задачи, в нём существует только папка `003` с `spec.md` и `plan.md`. Описание проекта и рабочие инструкции распределены между `PROJECT.md` и `AGENTS.md`, а централизованной навигации по знаниям проекта нет.

Issue требует превратить `memory-bank` в основное место для накопления проектового контекста, вынести туда базовые документы о проекте и разработке, а `AGENTS.md` сократить до минимальной точки входа.

## Цель
Собрать внутри `memory-bank` минимальную, но связанную структуру документации, которая:
- содержит общее описание проекта;
- объясняет, как собирать и тестировать решение;
- даёт единый индекс со ссылками на все документы;
- переводит существующий материал по задаче `003` в новую иерархию `features`.

## Что требуется реализовать
1. Создать раздел `memory-bank/project/` с базовой документацией:
   - `overview.md` — общее описание проекта;
   - `build.md` — как собирать проект;
   - `testing.md` — как устроены тесты и как их запускать.
2. Создать `memory-bank/index.md` как корневую точку входа в memory-bank.
   В индексе должны быть аннотированные ссылки на документы, созданные по этой задаче в разделе `project/`. Индексировать содержимое `features/`, включая перенесённые материалы по `003`, не требуется.
   Пример формата аннотированной ссылки:
   ```md
   - [путь/к/файлу.md](путь/к/файлу.md)
     **Что:** 1-2 предложения о содержимом файла.
     **Читать, чтобы:** когда и зачем агенту нужно сюда заходить.
   ```
3. Переложить существующую папку `memory-bank/003` в `memory-bank/features/003`.
   При переносе нужно сохранить содержимое файлов `spec.md` и `plan.md`.
4. Обновить корневой `AGENTS.md` так, чтобы в нём осталось только указание `читай memory-bank/index.md`.

## Требования к содержимому
- `project/overview.md` должен описывать назначение библиотеки, целевую платформу, основную доменную задачу и ключевые публичные сущности проекта.
- `project/build.md` должен опираться на реальные команды и структуру репозитория. В документе должно быть явно указано, что сборка проекта выполняется из командной строки средствами стека проекта. Если отдельного build-скрипта нет, это должно быть явно отражено.
- `project/testing.md` должен описывать тестовый проект, поддерживаемые сценарии тестирования и команду запуска тестов.
- `index.md` должен быть пригоден как стартовый файл для человека или агента, который впервые заходит в репозиторий.

## Ограничения и решения
- Нужно опираться на фактическое текущее состояние репозитория, а не на предполагаемую структуру.
- Папка `src/` является канонической директорией с исходным кодом и проектами; ошибочное упоминание `/scr` из `AGENTS.md` не должно механически переноситься в новую документацию без проверки.
- После сокращения `AGENTS.md` проект не должен терять доступ к важному контексту: он должен быть перенесён или переосмыслен внутри `memory-bank`.

## Предлагаемая структура после изменений
```text
memory-bank/
  index.md
  project/
    overview.md
    build.md
    testing.md
  features/
    003/
      spec.md
      plan.md
  007/
    spec.md
```

## Критерии приёмки
- В репозитории создан `memory-bank/project/` с файлами `overview.md`, `build.md`, `testing.md`.
- Создан `memory-bank/index.md` с аннотированными ссылками на:
  - `project/overview.md`;
  - `project/build.md`;
  - `project/testing.md`.
- Папка `memory-bank/003` отсутствует, а её содержимое перенесено в `memory-bank/features/003` без потери файлов.
- Содержимое `AGENTS.md` сведено к единственной инструкции: `читай memory-bank/index.md`.
- Содержимое новых документов согласовано с фактическим устройством репозитория и не противоречит `PROJECT.md`.

## Риски и открытые вопросы
- Если в существующих документах обнаружатся противоречия по структуре проекта или командам, их придётся разрешить при переносе, а не копировать дословно.
