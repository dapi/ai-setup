# Spec: Настройка CI/CD

**Feature:** 003 — Добавить CI/CD
**Issue:** akoltun/ai-setup#3
**Date:** 2026-04-06

---

## Обзор

Настраивается GitHub Actions CI-пайплайн с джобом `setup` и четырьмя зависящими от него джобами (`lint`, `compile`, `build`, `test`). Джоб `setup` подготавливает среду и устанавливает зависимости; остальные джобы получают готовый `node_modules` через артефакт и запускаются параллельно. Тестовое покрытие unit-тестов рассчитывается автоматически через встроенный механизм Bun. Порог покрытия задаётся в `bunfig.toml` (см. раздел «bunfig.toml» и «Текущее состояние покрытия»). Merge блокируется при любой ошибке через Branch Protection Rules. `README.md` актуализируется: все разделы приводятся в соответствие с текущим состоянием проекта, добавляется CI-бейдж.

---

## Не входит в scope

- Настройка деплоя и CD (только CI)
- Branch Protection Rules — ручной пост-мерж шаг, не входит в Definition of Done
- Повышение тестового покрытия выше baseline
- Настройка секретов GitHub — на текущем этапе не требуется

## Изменяемые файлы

### 1. `.github/workflows/ci.yml`

Заменить существующий файл новым пайплайном.

**Триггеры:**

- `push` в любую ветку
- `pull_request` (любая ветка)
- `workflow_dispatch` (ручной запуск)

**Concurrency:** отменять предыдущий запуск той же ветки (`cancel-in-progress: true`).

**Джобы:**

#### Джоб `setup`

Подготавливает среду и устанавливает зависимости. Все остальные джобы зависят от него (`needs: setup`).

Шаги:

1. `actions/checkout`
2. `oven-sh/setup-bun` — установить Bun
3. `bun install --frozen-lockfile`
4. `actions/upload-artifact` — загрузить `node_modules` как артефакт с именем `node-modules` и retention 1 день

#### Джобы `lint`, `compile`, `build`, `test` — запускаются параллельно на `ubuntu-latest`, зависят от `setup`

Общие шаги для каждого из этих джобов (перед специфичными шагами):

1. `actions/checkout`
2. `oven-sh/setup-bun` — установить Bun
3. `actions/download-artifact` — скачать артефакт `node-modules` в `./node_modules`

#### Джоб `lint`

Специфичные шаги:

1. `bun run lint:ci`

#### Джоб `compile`

Специфичные шаги:

1. `bun run compile`

#### Джоб `build`

Специфичные шаги:

1. `bun run build`

#### Джоб `test`

Тестирование в два этапа:

1. Unit-тесты с coverage (Bun Test Runner)
2. E2E-тесты (Playwright)

Специфичные шаги:

1. `bun run build` — сборка нужна для запуска Playwright web-сервера в production-режиме
2. `bunx playwright install --with-deps chromium` — установить браузер для Playwright
3. `bun test --coverage` — unit-тесты с расчётом покрытия; при покрытии ниже порога завершается с кодом ошибки
4. `bunx playwright test` — e2e-тесты

**Переменная окружения для Playwright:** `NODE_ENV=production`, чтобы web-сервер запускался командой `bun run serve` (использует готовый `dist/`).

**Временной лимит джоба `test`:** 30 минут.

### 2. `package.json`

Добавить новый скрипт `lint:ci` в секцию `"scripts"`:

```json
"lint:ci": "oxlint && oxfmt --check"
```

Отличие от `lint`: не применяет автоисправления (`--fix` и `--write` убраны). В CI файлы не должны изменяться.

---

### 3. `bunfig.toml`

Если файл не существует — создать. Если секция `[test]` уже существует — заменить её целиком.

Добавить секцию `[test]` для настройки coverage:

```toml
[test]
coverage = true
coverageThreshold = 0.8
```

`coverageThreshold = 0.8` — порог 80% по всем метрикам покрытия (line, function, statement). При запуске `bun test --coverage` процесс завершится с ненулевым кодом, если хотя бы одна метрика ниже 80%.

**Примечание:** начальное значение `coverageThreshold` — `0.8`. Если после запуска CI фактическое покрытие окажется ниже 80% (см. раздел «Текущее состояние покрытия»), агент должен снизить значение до фактического покрытия и зафиксировать его как baseline.

## Поведение при ошибках инфраструктуры

- При сбое `actions/upload-artifact` в джобе `setup`: джоб `setup` завершается с ошибкой, зависимые джобы не запускаются.
- При сбое `actions/download-artifact` в зависимом джобе: джоб завершается с ошибкой.
- Ретраев нет — при любом сбое пайплайн падает целиком.

## Настройка репозитория (ручные шаги)

Эти шаги выполняются вручную после мержа CI-пайплайна в `main`. По завершении работы агент должен вывести на экран пошаговую инструкцию, включающую все пункты ниже.

### Branch Protection Rules

После того, как CI-пайплайн будет смержен в `main` и пройдёт хотя бы один раз, в настройках репозитория GitHub настроить Branch Protection Rule для ветки `main`:

- **Require status checks to pass before merging:** включить
- Добавить все пять статус-чеков: `setup`, `lint`, `compile`, `build`, `test`
- **Do not allow bypassing the above settings:** включить (опционально, но рекомендуется)

## Инварианты

1. `lint:ci` и `lint` используют одинаковый набор правил oxlint и oxfmt — отличается только режим (проверка vs автоисправление)
2. `coverageThreshold` в `bunfig.toml` ≤ фактического покрытия unit-тестов — CI не должен перманентно падать из-за завышенного порога
3. Зависимые джобы (`lint`, `compile`, `build`, `test`) не запускаются без успешного завершения `setup`

## Ограничения и решения

### Покрытие только unit-тестов

Порог 80% применяется только к unit-тестам (Bun Test Runner). Playwright e2e-тесты не вносят вклад в расчёт покрытия — это стандартная практика: e2e тестируют поведение приложения, а не покрытие кода.

### Текущее состояние покрытия

На момент написания спека в проекте один unit-тест (`src/domains/home/api.test.ts`). Целевой порог покрытия по Brief — 80%, однако при текущем количестве тестов фактическое покрытие, вероятно, будет ниже. Установка `coverageThreshold = 0.8` при фактическом покрытии, например, 40% приведёт к тому, что CI будет перманентно падать, блокируя любой merge. Поэтому, если после первого запуска CI покрытие окажется ниже 80%, агент должен снизить `coverageThreshold` в `bunfig.toml` до фактического значения покрытия и зафиксировать его как baseline. Это гарантирует, что CI проходит, а покрытие не снижается ниже текущего уровня. Повышение покрытия до 80% — отдельная задача.

### Lint в CI vs локально

Скрипт `lint` (для локальной разработки) применяет автоисправления. Скрипт `lint:ci` (для CI) только проверяет без изменений. Оба скрипта должны быть согласованы: если `lint:ci` падает, `lint` должен это исправить.
