# Plan: issue #3

## Цель
Добавить поддержку мэппинга вложенных JSON Pointer-путей для `path` и `from` в `Mapper`, сохранив текущие правила `same-name`, `MapFrom(...)`, `Ignore()` и преобразование значений.
Поддержка в рамках issue распространяется только на уже покрытые операции `add`, `replace`, `remove`, `test`; полноценную поддержку RFC-операций `move` и `copy` добавлять не требуется.

## План работ

### 1. Зафиксировать модель nested-resolve в рантайме
- Разделить текущую логику "получить правило по точному пути" и "разрешить полный nested-path".
- Ввести отдельную модель результата резолвинга пути, в которой будут:
  - итоговый `TargetPath`;
  - признак `Ignored`;
  - конвертер значения для leaf-сегмента;
  - метаданные leaf/source/target типов, нужные для проверки supported leaf-object сценариев;
  - достаточно данных для осмысленных сообщений об ошибках.
- Сделать так, чтобы `MapOperation(...)` работал через единый резолвер и одинаково обрабатывал `path` и `from`.
- Явно зафиксировать на уровне рантайма, что срабатывание `Ignore()` для `path` или для `from` приводит к пропуску операции целиком.

### 2. Добавить раннюю валидацию JSON Pointer
- Выделить общий валидатор для `path` и `from`, который выполняется до поиска правил.
- Считать `NotSupportedException` обязательным результатом для:
  - пустого пути;
  - пути без ведущего `/`;
  - пустых сегментов;
  - завершающего `/`;
  - массивов и индексов;
  - escape-последовательностей `~0` и `~1`.
- Проверку unsupported-path встроить раньше `Ignore()`, чтобы поведение соответствовало spec.

### 3. Расширить структуру скомпилированных правил
- Отказаться от предположения, что правило существует только для полного верхнеуровневого пути.
- Хранить в compiled map данные, достаточные для пошагового резолвинга:
  - явные правила `MapFrom(...)` по исходному пути;
  - default same-name правила по сегменту;
  - `Ignore()` по целевому пути;
  - типы source/target свойств для перехода к вложенному профилю.
- На этапе компиляции сохранить приоритеты:
  - явный `MapFrom(...)` выше nested-profile;
  - nested-profile выше implicit fallback;
  - silent fallback к исходному пути убрать.

### 4. Обновить компиляцию профиля
- Переработать `PatchMapCompiler.BuildDefaultRules(...)`, чтобы он строил не только top-level соответствия, но и данные по доступным target/source свойствам для root-segment resolve.
- В `ApplyConfiguration(...)` сохранить явные nested-path из `SourceMemberDescriptor`, а не ограничиваться верхним уровнем.
- Явно сохранить текущее ограничение на supported `MapFrom(...)`-выражения для nested-case:
  - выражение должно зависеть ровно от одного исходного поля или вложенного исходного поля;
  - multi-source выражения не расширять и не начинать поддерживать в рамках этой задачи.
- Добавить проверку конфликтов:
  - если два `ForMember(...).MapFrom(...)` приводят к одному и тому же исходному пути, выбрасывать `InvalidOperationException`;
  - в сообщении указывать конфликтующий путь и пару типов.
- Сохранить текущее поведение `Ignore()` для top-level и nested target path.

### 5. Реализовать рекурсивный резолвинг пути
- Разбить исходный путь на сегменты и резолвить его слева направо.
- Для каждого шага:
  - попытаться найти полное явное правило `MapFrom(...)` для уже известного исходного пути;
  - если явного правила нет, разрешить текущий сегмент через same-name/default mapping;
  - если остаётся remainder, определить типы промежуточных source/target свойств и запросить вложенный compiled map.
- Для перехода между типами использовать уже существующий кэш `ConcurrentDictionary<TypePair, ICompiledMap>` в `Mapper`, а не дублировать регистрацию профилей.
- Если вложенный профиль обязателен, но отсутствует, выбрасывать `InvalidOperationException` с парой типов.
- Если путь не сматчился на текущем уровне, выбрасывать `InvalidOperationException` с проблемным исходным путём и парой типов.

### 6. Ограничить поддержку только leaf-свойствами из spec
- После успешного резолвинга проверять, что конечное свойство является допустимым leaf-типом:
  - `bool`;
  - `string`;
  - enum;
  - числовой тип.
- Если операция адресует вложенный объект целиком, выбрасывать `NotSupportedException`.
- Эту проверку применять и к `path`, и к `from`.

### 7. Сохранить корректное преобразование значений
- Для `add`/`replace`/`test` продолжить использовать конвертацию только по leaf-правилу конечного пути.
- Убедиться, что nested `MapFrom(...)` от одного исходного поля по-прежнему даёт корректный `ValueConverter`.
- Не вводить конвертацию для `remove`.
- Не расширять поддержку операций за пределы `add`, `replace`, `remove`, `test`; `move` и `copy` оставить вне scope текущей реализации.

### 8. Адаптировать рантайм `Mapper`
- Передать compiled map доступ к реестру всех compiled maps, чтобы nested-resolve мог переходить к другим парам типов.
- Избежать циклической рекурсии в момент конструирования: compiled map должен ссылаться на resolver/provider, а не на уже собранные nested-объекты напрямую.
- Сохранить публичный API `Mapper` без изменений.

### 9. Добавить отдельный набор nested-тестов
- Создать новый файл тестов, например `src/Mapper.Tests/NestedMapperTests.cs`.
- Покрыть позитивные сценарии:
  - same-name nested path;
  - глубина больше одного уровня;
  - rename через `MapFrom(...)`;
  - nested `from`;
  - `Ignore()` для `from` с пропуском операции целиком;
  - `Ignore()` на верхнем target field;
  - `Ignore()` на вложенном target field;
  - конвертация значения при отличии типов;
  - вычисляемое преобразование из одного вложенного исходного поля;
  - приоритет явного `MapFrom(...)` над nested-profile;
  - сценарий, где явное nested `MapFrom(...)` не требует промежуточного профиля.
- Покрыть негативные сценарии:
  - отсутствующий вложенный профиль;
  - несматченный root segment;
  - несматченный leaf segment;

- В проверках исключений дополнительно валидировать содержание сообщения: проблемный путь и типовую пару.

## Риски
- Основной риск в смешении двух разных режимов резолвинга: явный full-path `MapFrom(...)` и пошаговая рекурсия через вложенные профили. Их приоритет нужно зафиксировать тестами до рефакторинга.
- Второй риск в том, что текущий `Ignore()` привязан к поиску по `TargetPath`; после расширения на nested-path легко потерять часть сценариев, если хранить только source-index.
- Третий риск в том, что compiled map сейчас полностью автономен. Для nested-resolve ему понадобится доступ к другим type-pair mappings без поломки текущего lazy/runtime поведения.
