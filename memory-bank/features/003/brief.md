# Feature 3: Role-based operational portals (Manager & Staff)

## 1. Задача

Реализовать ролевую модель доступа и отдельные рабочие зоны (namespaces) для пользователей системы: `manager` и `staff`, с соответствующими правами для работы с тикетами и сотрудниками.

Цель — отделить:

* административную часть (admin),
* операционное управление (manager),
* исполнительский уровень (staff),

и обеспечить каждому уровню доступ только к необходимому функционалу.

---

## 2. Проблема

На текущий момент:

* существует только admin-уровень управления;
* отсутствует разграничение доступа для операционных ролей;
* нет изолированных интерфейсов для manager и staff;
* работа с тикетами не разделена по ролям (assign / execution).

Это приводит к:

* перегруженности admin-интерфейса;
* отсутствию безопасного делегирования задач;
* невозможности масштабирования системы на реальные команды (manager + staff).

---

## 3. Контекст

Система представляет собой ticketing/операционную платформу в контексте отелей.

Уже реализовано:

* admin namespace;
* базовые сущности (staff, departments, tickets);
* read-only доступ к staff (частично глобальный и hotel-scoped).

Текущая фича расширяет систему до полноценной role-based модели с отдельными интерфейсами.

---

## 4. Для кого это делается

### Admin

* управляет системой глобально;
* создает структуру (staff, departments, hotels).

### Manager

* управляет операционной работой в рамках своего отеля;
* распределяет тикеты;
* управляет сотрудниками (без изменения ролей).

### Staff

* выполняет задачи;
* работает только со своими тикетами.

---

## 5. Scope (что входит в фичу)

### 5.1. Архитектура

Вводятся отдельные namespaces:

* `/admin/...`
* `/manager/...`
* `/staff/...`

С разграничением доступа по ролям.

---

### 5.2. Manager функциональность

Manager работает в рамках своего отеля.

#### Staff management

* просмотр списка staff
* создание staff
* редактирование staff:

    * name
    * email
    * password
* без возможности:

    * менять role
    * удалять staff
    * деактивировать staff (вне scope)

#### Tickets

* просмотр всех tickets отеля
* фильтрация по department
* просмотр tickets конкретного department
* назначение ticket:

    * на department
    * на конкретного staff
* изменение статуса ticket
* добавление комментариев

#### Departments

* только просмотр списка departments
* без CRUD

---

### 5.3. Staff функциональность

Staff работает только со своими assigned tickets.

* просмотр списка своих tickets
* просмотр конкретного ticket
* изменение статуса ticket
* добавление комментариев

Ограничения:

* нет доступа к чужим tickets
* нет доступа к staff management
* нет доступа к departments

---

## 6. Ограничения

* manager и staff не имеют доступа к `/admin` namespace
* manager не может:

    * менять role пользователей
    * удалять или деактивировать staff
* staff работает только со своими tickets
* доступ manager ограничен его отелем
* доступ staff ограничен назначенными ему tickets
* comments являются стандартными ticket-комментариями (не internal-only)

---

## 7. Out of scope

* деактивация staff
* удаление staff
* CRUD для departments со стороны manager
* сложная ролевая система (ACL, permissions matrix)
* audit log
* notification system

---

## 8. Связь с другими фичами

* зависит от:

    * Staff CRUD (admin)
    * Department CRUD (admin)
* расширяет:

    * ticket workflow
* подготавливает систему к:

    * масштабированию команд
    * multi-role доступу
    * дальнейшему RBAC
