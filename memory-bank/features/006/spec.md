# Spec: issue #6

## Источник
- GitHub issue: https://github.com/atamanuk/mapper/issues/6
- Заголовок: `Демонстрационный проект с возможностью опробовать мэппинг`
- Статус: `OPEN`

## Контекст
Сейчас репозиторий содержит только библиотеку `Mapper` и тестовый проект `Mapper.Tests`; отдельного demo-приложения для ручной проверки мэппинга нет. В issue требуется добавить одностраничный веб-проект `Mapper.Demo` на `dotnet` + Razor Pages, расположенный рядом с основной библиотекой, чтобы можно было вводить Json Patch для source-модели и получать результат мэппинга на backend.

Для данной задачи формат отображения моделей в demo фиксируется как `name: type`, а не JSON Schema. Это сознательное упрощение UI для первого демонстрационного сценария.

По фактическому состоянию репозитория solution-файл сейчас один: `src/Mapper.sln`, который включает `Mapper` и `Mapper.Tests`.

Для этой задачи рабочим solution считается `src/Mapper.sln`. Новый demo-проект должен быть подключён именно к нему, а документация должна использовать его как каноническую точку сборки и запуска.

## Цель
Добавить в репозиторий минимальный, но рабочий демонстрационный веб-проект `Mapper.Demo`, который:
- показывает пользователю описания полей source- и target-моделей в формате `name: type`;
- принимает Json Patch-операции для source-модели;
- по кнопке `Map` вызывает backend-мэппинг через библиотеку `Mapper`;
- выводит результат мэппинга или диагностическую ошибку в UI.

## Что требуется реализовать
1. Создать новый проект `Mapper.Demo` на ASP.NET Core Razor Pages.
   Проект должен находиться рядом с библиотекой в директории `src/`, а не вне основной структуры исходников.
2. Подключить `Mapper.Demo` к `src/Mapper.sln`.
   После этого проект должен собираться стандартной командой `dotnet build src/Mapper.sln` без ручного добавления проекта пользователем.
   Проект также должен запускаться локально командой `dotnet run --project src/Mapper.Demo/Mapper.Demo.csproj`.
   Для удобного локального запуска из редактора нужно оформить `.vscode/launch.json` с конфигурацией, которая без ручного редактирования запускает `src/Mapper.Demo/Mapper.Demo.csproj`.
3. Реализовать одну web-страницу без авторизации с набором элементов из issue:
   - блок с описанием полей модели `Source` в формате `name: type`;
   - блок с описанием полей модели `Target` в формате `name: type`;
   - текстовое поле для ввода Json Patch над `Source`;
   - кнопку `Map`;
   - текстовое поле или аналогичный блок для результата мэппинга.
4. Реализовать backend-обработчик, который:
   - принимает введённый Json Patch;
   - десериализует его в `JsonPatchDocument<TSource>`;
   - прогоняет через `Mapper`;
   - сериализует результат мэппинга как `JsonPatchDocument<TTarget>` обратно в JSON для отображения на странице.
5. Добавить демонстрационные модели и профиль мэппинга, достаточные для показа ограниченного demo-сценария этой задачи.
   Для demo нужно зафиксировать конкретный сценарий:
   - `Source` содержит поля `Name: string` и `Age: int`;
   - `Target` содержит поля `DisplayName: string` и `Age: int`;
   - поле `Age` демонстрирует same-name mapping;
   - поле `Name` маппится в `DisplayName` через `MapFrom(...)`.
   Дополнительные сценарии, кроме same-name mapping и `MapFrom(...)`-переименования, в эту задачу не входят.
6. Обработать пользовательские ошибки в demo:
   - невалидный JSON во входном patch;
   - ошибки мэппинга/конфигурации;
   - пустой ввод.
   Ошибки должны отображаться на странице в читаемом виде, а не только приводить к HTTP 500.
7. Обновить и синхронизировать документацию репозитория с фактической реализацией demo так, чтобы было понятно:
   - где находится `Mapper.Demo`;
   - как его запускать;
   - что именно демонстрирует страница.
   Обязательная точка синхронизации документации: `memory-bank/project/build.md`, где должна быть актуальная информация про сборку и запуск demo-проекта.

## Ожидаемое поведение
- При открытии demo-страницы пользователь сразу видит описания полей `Source` и `Target` в формате `name: type`, а поле ввода patch-операций предзаполнено валидным примером.
- При отправке валидного patch backend возвращает JSON сериализованного `JsonPatchDocument<TTarget>`, и этот JSON отображается в результирующем блоке без перезагрузки на отдельную страницу результата.
- Если входной patch использует поле source-модели, которое переименовывается профилем, в результате должен появиться path к соответствующему полю target-модели.
- При невалидном вводе пользователь получает понятное сообщение об ошибке и не теряет введённый patch.
- Стартовый пример patch-а должен быть таким:
  ```json
  [
    { "op": "replace", "path": "/Name", "value": "Alice" },
    { "op": "replace", "path": "/Age", "value": 42 }
  ]
  ```
- Для этого примера demo должен показывать результат:
  ```json
  [
    { "op": "replace", "path": "/DisplayName", "value": "Alice" },
    { "op": "replace", "path": "/Age", "value": 42 }
  ]
  ```

## Требования к реализации
- `Mapper.Demo` должен ссылаться на локальный проект `Mapper`, а не на опубликованный NuGet-пакет.
- Технологический стек проекта должен соответствовать issue: `dotnet core` и Razor Pages.
- Страница должна быть одностраничным demo-сценарием; отдельная система авторизации или многостраничный интерфейс не нужны.
- Реализация не должна менять публичный API библиотеки только ради demo, если текущих возможностей `Mapper` уже достаточно.
- Demo должен использовать тот же стек JSON Patch, который фактически используется библиотекой `Mapper` в репозитории на момент реализации задачи; нельзя закладываться на другой пакет или другой формат patch-документа без явного обновления зависимости и документации в рамках этой же задачи.
- Описания моделей в UI должны выводиться в формате `name: type`, генерироваться из реальных demo-типов или поддерживаться в коде в явной синхронизации с ними; нельзя показывать произвольный текст, не связанный с фактическими моделями.
- В текущей задаче demo не должен добавлять отдельные сценарии сверх same-name mapping и `MapFrom(...)`-переименования.
- В репозитории должен быть `.vscode/launch.json` с конфигурацией запуска `src/Mapper.Demo/Mapper.Demo.csproj` без ручного редактирования пользователем.

## Технические наблюдения по текущему репозиторию
- Основная библиотека находится в `src/Mapper/Mapper.csproj`.
- Тесты находятся в `src/Mapper.Tests/Mapper.Tests.csproj`.
- В библиотеке уже используется пакет `Microsoft.AspNetCore.JsonPatch`; demo должен быть совместим с тем способом представления и обработки patch-документов, который фактически поддерживает текущая реализация `Mapper`.
- В репозитории используется `src/Mapper.sln`; документацию после добавления demo нужно синхронизировать с этим способом сборки и запуска.

## Критерии приёмки
- В `src/` создан проект `Mapper.Demo` на Razor Pages.
- Новый проект подключён к `src/Mapper.sln`, и solution собирается командой `dotnet build src/Mapper.sln` без ручных шагов добавления проекта.
- `Mapper.Demo` запускается локально командой `dotnet run --project src/Mapper.Demo/Mapper.Demo.csproj`.
- В `.vscode/launch.json` есть конфигурация, которая без ручного редактирования запускает `src/Mapper.Demo/Mapper.Demo.csproj`.
- `Mapper.Demo` имеет ссылку на локальный `Mapper.csproj` и использует библиотеку для реального мэппинга patch-документов.
- На главной странице отображаются описания полей `Source` и `Target` в формате `name: type`, поле ввода patch, кнопка `Map` и блок результата.
- Поле ввода при первой загрузке страницы предзаполнено обязательным примером patch-а из раздела "Ожидаемое поведение".
- Демонстрационный сценарий показывает минимум:
  - same-name mapping для `Age -> Age`;
  - `MapFrom(...)` с переименованием `Name -> DisplayName`.
- При корректном patch пользователь получает валидный JSON сериализованного `JsonPatchDocument<TTarget>`.
- При пустом вводе пользователь получает читаемое сообщение об ошибке на странице без падения приложения и без потери введённого текста.
- При невалидном JSON пользователь получает читаемое сообщение об ошибке на странице без падения приложения и без потери введённого текста.
- При ошибке мэппинга или конфигурации пользователь получает читаемое сообщение об ошибке на странице без падения приложения и без потери введённого текста.
- В `memory-bank/project/build.md` актуально описаны сборка и запуск demo-проекта, а также указано, что он входит в `src/Mapper.sln`.

## Риски и открытые вопросы
- Описания полей в формате `name: type` должны оставаться синхронизированными с demo-моделями; если они не генерируются автоматически, нужно выбрать простую явную точку их обновления в коде.
- Если текущая библиотека не предоставляет удобной обвязки для десериализации patch-документа в demo, эту интеграцию придётся аккуратно собрать в самом web-проекте без расширения объёма задачи до полноценного UI/API-фреймворка.
