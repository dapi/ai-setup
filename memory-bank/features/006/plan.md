# Plan: issue #6

## Цель
Добавить в `src/` демонстрационный проект `Mapper.Demo` на Razor Pages, подключённый к `src/Mapper.sln`, чтобы можно было вручную вводить `Json Patch` для `Source`, прогонять его через библиотеку `Mapper` и видеть сериализованный результат для `Target` или читаемую ошибку.

## Основные точки изменений
- `src/Mapper.sln`
- Новый проект `src/Mapper.Demo/`
- `.vscode/launch.json`
- `memory-bank/project/build.md`
- Дополнительные документы `memory-bank`, если для синхронизации запуска и назначения demo этого потребует реализация

## Шаги реализации

### 1. Подтвердить фактическую базу для demo
- Проверить текущую структуру `src/Mapper.sln`, состав проектов и способ запуска через `dotnet`.
- Уточнить, каким пакетом и API в репозитории уже представлен `JsonPatchDocument`, чтобы demo использовал тот же стек, что и библиотека.
- Найти минимальную точку интеграции с `Mapper`, не меняя публичный API библиотеки только ради demo.

### 2. Создать проект `Mapper.Demo`
- Добавить новый ASP.NET Core Razor Pages проект в `src/Mapper.Demo`.
- Подключить его к `src/Mapper.sln`.
- Настроить ссылку на локальный проект `src/Mapper/Mapper.csproj`, а не на внешний пакет.
- Сразу проверить, что solution собирается командой `dotnet build src/Mapper.sln`, а demo запускается командой `dotnet run --project src/Mapper.Demo/Mapper.Demo.csproj`.

### 3. Зафиксировать demo-сценарий в коде
- Добавить demo-модели:
  - `Source` с полями `Name: string` и `Age: int`;
  - `Target` с полями `DisplayName: string` и `Age: int`.
- Добавить профиль мэппинга, который демонстрирует:
  - same-name mapping для `Age -> Age`;
  - `MapFrom(...)`-переименование `Name -> DisplayName`.
- Выбрать простую точку, из которой UI будет получать описания полей в формате `name: type`, чтобы они не расходились с реальными типами.

### 4. Реализовать backend-обработчик страницы
- Сделать одну Razor Page без авторизации как основной экран demo.
- Добавить обработчик, который:
  - принимает текст введённого patch;
  - валидирует пустой ввод;
  - десериализует JSON в `JsonPatchDocument<Source>`;
  - вызывает мэппинг через библиотеку `Mapper`;
  - сериализует результат как `JsonPatchDocument<Target>` обратно в JSON для отображения.
- Не уводить пользователя на отдельную страницу результата; ответ должен оставаться на той же странице.

### 5. Собрать UI одностраничного demo
- Разместить на главной странице:
  - блок описания полей `Source`;
  - блок описания полей `Target`;
  - поле ввода Json Patch;
  - кнопку `Map`;
  - блок результата.
- Предзаполнить поле ввода обязательным примером:
  ```json
  [
    { "op": "replace", "path": "/Name", "value": "Alice" },
    { "op": "replace", "path": "/Age", "value": 42 }
  ]
  ```
- Обеспечить отображение ожидаемого результата с путём `/DisplayName` после отправки валидного patch.

### 6. Обработать ошибки пользователя и runtime-ошибки
- Явно обрабатывать:
  - пустой ввод;
  - невалидный JSON;
  - ошибки мэппинга;
  - ошибки конфигурации.
- Показывать сообщения об ошибках на странице в читаемом виде без HTTP 500.
- Сохранять введённый пользователем patch в форме после любой ошибки.

### 7. Добавить локальный запуск из редактора
- Создать `.vscode/launch.json`.
- Добавить конфигурацию, которая без ручного редактирования запускает `src/Mapper.Demo/Mapper.Demo.csproj`.
- Проверить, что конфигурация не конфликтует с уже существующими файлами workspace.

### 8. Синхронизировать документацию
- Обновить `memory-bank/project/build.md`, чтобы там было явно указано:
  - что `Mapper.Demo` находится в `src/Mapper.Demo`;
  - что demo включён в `src/Mapper.sln`;
  - как собрать solution;
  - как запустить demo.
- В документации должно быть явно описано, что именно демонстрирует страница:
  - ввод `Json Patch` для `Source`;
  - мэппинг patch-документа через библиотеку `Mapper`;
  - вывод результата как `JsonPatchDocument<TTarget>`;
  - ограниченность demo-сценария примерами `Age -> Age` и `Name -> DisplayName`.
- При необходимости кратко обновить другие документы `memory-bank`, если без этого назначение demo, точка запуска или содержание demo-сценария остаются неочевидными.

### 9. Провести итоговую валидацию
- Прогнать сборку `dotnet build src/Mapper.sln`.
- По возможности проверить запуск demo, стартовое состояние страницы и happy-path сценарий с примером из spec.
- Убедиться, что:
  - проект подключён к solution;
  - локальная ссылка на `Mapper` используется корректно;
  - при первой загрузке страница сразу показывает поля `Source` и `Target` в формате `name: type`;
  - при первой загрузке поле ввода уже предзаполнено обязательным примером patch-а из spec;
  - валидный patch маппится в `Target` и для примера из spec даёт сериализованный результат с путями `/DisplayName` и `/Age`;
  - пустой ввод даёт читаемую ошибку на странице и не очищает введённый текст;
  - невалидный JSON даёт читаемую ошибку на странице и не очищает введённый текст;
  - ошибка мэппинга или конфигурации даёт читаемую ошибку на странице и не очищает введённый текст;
  - документация соответствует фактической структуре репозитория.

## Порядок выполнения
1. Сначала проверить текущую интеграцию `Mapper` и стек `JsonPatch`.
2. Затем создать и подключить `Mapper.Demo`.
3. После этого реализовать demo-модели, профиль, backend и UI страницы.
4. Затем добавить `.vscode/launch.json`.
5. В конце синхронизировать документацию и прогнать валидацию.

Такой порядок снижает риск переделки UI до того, как подтверждён способ реального вызова `Mapper` и сборка проекта внутри `src/Mapper.sln`.

## Риски
- Основной риск в том, что demo должен использовать тот же `Json Patch` стек, что и библиотека; если в web-проекте подключить другой способ сериализации или другой пакет, демонстрация будет расходиться с реальным поведением `Mapper`.
- Второй риск в десериализации `JsonPatchDocument<Source>` внутри Razor Pages: нужно аккуратно выбрать формат и сериализацию, чтобы пользовательский ввод корректно превращался в patch-документ и обратно в JSON результата.
- Третий риск в синхронизации описаний `name: type` с demo-моделями: если UI будет хранить эти строки отдельно от типов, их легко рассинхронизировать при будущих изменениях.
